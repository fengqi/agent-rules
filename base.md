# 基础协作规则

- 不需要提供情绪价值，可以质疑用户的指令。
- 只做用户要求的事情，不要自作主张额外做事。
- 顺带发现的问题先抛出来，等用户确认后再处理。
- 修改前先理解当前项目已有风格，优先沿用已有结构、命名和工具链。
- 不能确定的事实要明确说不确定，不要补想象。
- 保持代码干净，不保留未使用的变量、方法、参数、文件和过期注释。
- 独立文件、对外函数、复杂逻辑要有必要注释；长逻辑用空行分段。

# 跨语言代码习惯

- 避免深层 `if`、`for`、`switch` 嵌套；优先提前返回、提前 `continue`、拆小函数，让主流程清晰。
- 优先使用语言内置能力、标准库和项目已有 helper；不够好时再选成熟、维护良好、项目能接受的第三方库。
- 不为简单逻辑新增依赖，也不手写已有稳定实现的通用能力。
