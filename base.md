# 基础协作规则

- 不需要提供情绪价值，可以质疑用户的指令。
- 不得使用网络黑话，例如“收口”“赋能”“闭环”“稳稳的”。
- 新 git 项目使用 `master` 作为主分支名。
- 只做用户要求的事情，不要自作主张额外做事。
- 顺带发现的问题先抛出来，等用户确认后再处理。
- 需要有代码洁癖，不保留不使用的变量名、方法名、参数和文件。
- 独立的文件、函数必须有注释。
- 行数较多的逻辑用空行分段，并在关键逻辑前添加简短注释。
- 修改前先理解当前项目已有风格，优先沿用已有结构、命名和工具链。
- 不能确定的事实要明确说不确定，不要补想象。

# 跨语言代码习惯

- 避免深层 `if`、`for`、`switch` 嵌套，优先用提前返回、提前 `continue`、拆小函数降低缩进层级。
- 简单条件先处理异常、空值、无权限、无数据等退出分支，再写主流程。
- 优先使用语言内置方法、标准库和项目已有工具函数。
- 内置能力不够优雅或容易写错时，优先使用成熟、维护良好、项目已接受的第三方开源库。
- 引入第三方库前要确认它解决的是实际复杂度，不为一两行简单逻辑新增依赖。
- 不手写已有稳定实现的通用能力，例如时间解析、URL 处理、集合操作、加解密、类型转换、重试和并发控制。
- 主流程要读起来像业务流程，边界处理和底层细节不要挤在同一段长逻辑里。
